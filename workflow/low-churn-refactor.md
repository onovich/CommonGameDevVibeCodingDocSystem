# 低 Churn 重构方法论

**适用场景**: refactor

## 核心原则

重构的目标是改善结构，而不是制造噪音。低 churn 意味着：
- 最小化 diff 行数
- 最小化文件改动数
- 最小化 token 消耗
- 最大化可验证性

## 方法

### 1. 一次只做一个 slice

每个 slice 应该：
- 只搬运一个职责
- 只涉及 1-3 个文件
- 改完立即验证
- 验证通过后停止

超过 5 个文件通常说明 scope 膨胀。

### 2. 固定 slice 模板

每次改造前先输出：

```
Scope
- 这次只改哪一个职责
- 只涉及哪几个文件

Invariants
- 哪些行为必须保持不变
- 哪些 API 不能破坏

Planned responsibility move
- 把哪一段职责从 A 搬到 B

Verification
- 改完做哪一步验证

Stop boundary
- 到哪里必须停
- 哪些"顺手优化"明确不做
```

### 3. 禁止"顺手优化"

常见陷阱：
- 重构时顺手改玩法
- 搬运职责时顺手格式化
- 拆分类时顺手加新功能
- 收口边界时顺手重命名无关字段

正确做法：
- 架构 slice 期间不改玩法
- 格式化单独做
- 新功能单独做
- 重命名单独做

### 4. 验证优先

- 每个 slice 完成后立即验证
- 不要攒多个 slice 再验证
- 验证失败时容易定位问题
- 验证通过后立即停止

## 常见弯路

### 弯路 1：一开始就想一次性改成"理想架构"

问题：
- scope 爆炸
- token 浪费
- 编译失败后很难定位

正确做法：
- 只选一个 slice
- 只搬一个职责
- 立即验证

### 弯路 2：为了"解耦"到处加事件

问题：
- 事件链会掩盖真实依赖
- 对单机运行时结构，不一定比明确调用更好

正确做法：
- 逻辑层直接调用明确的边界
- 保持调用路径短、可追踪

### 弯路 3：重构时顺手改玩法

问题：
- 难以判断问题来自架构迁移还是玩法变更

正确做法：
- 架构 slice 期间尽量不改玩法规则
- 玩法变更单独做

## 与其他方法的关系

- 配合"验证优先工作流"使用
- 配合"slice-based refactor"使用
- 配合"薄 skill + 厚文档"模式使用
