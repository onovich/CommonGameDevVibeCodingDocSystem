# 验证优先工作流

**适用场景**: prototype, production, refactor

## 核心原则

先验证，再扩 scope。

## 为什么验证优先

### 问题场景
- 写了大量代码后才发现编译失败
- 改了多个文件后才发现逻辑错误
- 攒了多个功能后才发现某个环节有问题

### 验证优先的优势
- 快速反馈：问题立即暴露
- 容易定位：只有一个变更点
- 降低回退成本：失败时只需回退一小步
- 降低认知负担：不需要同时追踪多个变更

## 验证节奏

### 最小节奏
1. 写代码
2. 立即验证
3. 通过后再继续

### 不要
- 写完多个功能再验证
- 改完多个文件再验证
- 攒多个变更再验证

## 验证类型

### 编译验证
- 新增文件后立即编译
- 修改接口后立即编译
- 不要等到"差不多了"再编译

### 运行验证
- 新增功能后立即运行
- 修改逻辑后立即运行
- 不要等到"全部完成"再运行

### 测试验证
- 有测试时立即跑测试
- 不要等到"觉得没问题了"再跑

## 引擎特定验证

### Unity
- 新增 `.cs` 文件后必须先纯编译，再执行 Editor 方法
- 使用 batchmode 无头编译验证
- 不要依赖 Editor 内编译

### Godot
- 使用 `--headless --quit` 验证
- 不要依赖 Editor 内编译

### Bevy
- 使用 `cargo check` 快速验证
- 使用 `cargo test` 验证测试

## 与其他方法的关系

- 是"低 churn 重构"的基础
- 配合"slice-based refactor"使用
- 配合"AI 低 token 工作流"使用
