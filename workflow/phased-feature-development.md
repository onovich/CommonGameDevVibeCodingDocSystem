# 功能开发阶段化工作流

**适用场景**: production

## 核心思想

将功能开发拆分成多个阶段，每个关键阶段请求确认后再继续。

## 为什么需要阶段化

### 问题场景

- 一口气写完整个功能，发现方向错了
- 架构决策没对齐，返工成本高
- UI 设计没确认，白写了代码

### 解决方案

在关键节点停下来请求确认：
- 架构决策确认
- 代码骨架确认
- UI 设计确认

## 推荐阶段

### Phase 0: 理解需求

**目标**：弄清楚要做什么。

**需要回答的问题**：
- 核心行为是什么？
- 属于哪个游戏关切域？
- 需要跨系统通信吗？
- 有独立的生命周期吗？
- 需要配置数据吗？
- 有 UI 表现吗？

### Phase 1: 架构决策

**目标**：确定功能归属哪个模块，是否需要新建文件。

**决策树**：
```
这个功能属于哪个层？
│
├── 独立的游戏关切域，有独立生命周期和 Tick？
│   └── → 需要新建 XSystem
│
├── 多个系统共享的能力（音效/UI/存档）？
│   └── → 归属已有 App，或新建 XApp
│
├── 某个已有 System 内部的新逻辑分支？
│   ├── 逻辑量小（1~3 个方法）？
│   │   └── → 在已有 Domain 中添加方法
│   └── 逻辑量大，可单独命名？
│       └── → 新建 XDomain 类
│
└── 只是一个数据容器或集合管理？
    └── → 新建 XEntity 或 XRepo
```

**⚠️ 请求确认节点 #1**：
- 功能归属的层
- 计划新建的文件列表
- 计划修改的文件列表
- 跨系统通信方向

### Phase 2: 数据层设计

**目标**：设计 TM / Entity / Repo。

**输出**：
- TM 字段列表
- Entity 字段列表
- Repo 接口

### Phase 3: 代码骨架

**目标**：创建文件，写空方法签名。

**⚠️ 请求确认节点 #2**：
- 展示所有新建文件的骨架
- 展示所有修改文件的 diff

### Phase 4: UI 设计与实现

**目标**：设计 UI 布局和交互。

**⚠️ 请求确认节点 #3**：
- UI 布局草图
- 交互流程

### Phase 5: 业务逻辑实现

**目标**：填充 Domain 方法。

### Phase 6: 装配接入

**目标**：在 Main.cs 中装配新模块。

### Phase 7: 资产配置

**目标**：创建配置资产。

### Phase 8: 审查与提交

**目标**：验证、审查、提交。

## 与其他方法的关系

- 配合"verification-first workflow"使用
- 配合"slice-based refactor"使用
