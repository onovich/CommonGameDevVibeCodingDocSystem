# AI 低 Token 工作流

**适用场景**: prototype, production, refactor

## 核心原则

在 AI 辅助开发中，token 是成本。低 token 工作流的目标是：
- 减少不必要的文件读取
- 减少重复的代码生成
- 减少无效的探索

## 方法

### 1. 按需读取，不全读

**不要**：
- 一上来读取整个 docs/
- 一上来 grep 全仓
- 一上来读取所有相关文件

**应该**：
- 先读索引文档，确定需要读哪些
- 只读当前 slice 直接相关的文件
- 碰到具体问题时再查具体文档

### 2. 文档拆分策略

**问题场景**：
- 单个文档过大，AI 一次读取消耗大量 token
- 文档混杂多个主题，AI 难以定位需要的部分

**解决方案**：
- 按主题拆分文档
- 每个文档聚焦一个主题
- 使用索引文档串联
- 文档内部使用纯文本路径引用，不用 Markdown 超链接

**推荐结构**：
```
docs/
├── INDEX.md              # 唯一索引入口
├── architecture.md       # 架构定义
├── workflow.md          # 工作流
├── refactor-playbook.md # 重构手册
└── lessons.md           # 经验教训
```

### 3. 薄 skill + 厚文档模式

**问题场景**：
- skill 里塞大段知识，导致 skill 本身成为高 token 入口
- 文档里没有调度逻辑，AI 不知道该读什么

**解决方案**：
- skill 只负责：阅读顺序、切分粒度、验证节奏、输出格式
- 文档负责：长期知识、架构定义、代码锚点、反模式映射

**skill 应该写什么**：
```markdown
1. 先读：docs/xxx.md
2. 再按需读：docs/yyy.md
3. 只读取当前 slice 直接相关的代码
4. 输出固定格式：Scope / Invariants / ...
5. 一次只允许推进一个 slice
6. 验证时优先复用 /verify-xxx
```

**skill 不应该写什么**：
- 完整架构定义
- 详细代码示例
- 长篇方法论

### 4. 小步验证

- 每个 slice 完成后立即验证
- 不要攒多个 slice 再验证
- 验证失败时容易定位，不浪费 token

### 5. 固定输出格式

使用固定的短格式输出，避免冗长总结：

```
Scope
- ...

Invariants
- ...

Verification result
- PASS/FAIL
```

不要：
- 长篇回顾
- 重复已经说过的内容
- 散文化总结

## 与其他方法的关系

- 配合"文档拆分策略"使用
- 配合"薄 skill + 厚文档"模式使用
- 配合"验证优先工作流"使用
- 配合"slice-based refactor"使用
