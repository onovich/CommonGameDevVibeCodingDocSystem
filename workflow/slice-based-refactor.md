# 基于 Slice 的渐进式重构

**适用场景**: refactor

## 什么是 Slice

Slice 是一个最小可验证的重构单元：
- 只搬运一个职责
- 只涉及少量文件
- 改完可以立即验证
- 验证通过后可以停止

## 为什么用 Slice

### 优势
- 降低风险：每步都可验证
- 降低成本：失败时容易回退
- 降低认知负担：一次只关注一个边界
- 降低 token 消耗：不需要一次性读全仓

### 对比一次性重构
| 维度 | Slice-based | 一次性重构 |
|------|-------------|-----------|
| 风险 | 低 | 高 |
| 可验证性 | 每步验证 | 最后验证 |
| 失败定位 | 容易 | 困难 |
| Token 消耗 | 低 | 高 |
| 回退成本 | 低 | 高 |

## 推荐 Slice 顺序

对于"原型到生产架构"的收敛，推荐顺序：

1. **收口入口与依赖**
   - 把启动、注入、绑定集中到一个地方
   - 让依赖关系显式化

2. **提炼共享上下文**
   - 把"无结构引用袋"收敛成明确的上下文对象
   - 明确共享依赖边界

3. **集中帧驱动**
   - 把分散的 Update/FixedUpdate/LateUpdate 收口
   - 明确每帧调用顺序

4. **下沉业务逻辑**
   - 把跨对象规则从 Manager/Controller 中剥离
   - 形成独立的业务层

5. **收口对象管理**
   - 把创建/回收/注册路径集中
   - 禁止散落的对象生命周期操作

6. **收口 UI 边界**
   - 逻辑层不直接操作具体 UI 组件
   - 经过明确的 UI 边界

7. **收口资源加载**
   - 配置读取集中
   - 避免宽泛扫描

## Slice 控制原则

### 大小控制
- 1 个 slice 最好只碰一个边界
- 优先改 1 到 3 个文件
- 超过 5 个文件时，通常说明 scope 已经膨胀

### 停止边界
- 改完立刻验证并停下
- 不要"顺手再改一点"
- 不要"既然都改了，不如把 XX 也改了"

### 验证节奏
- 每个 slice 完成后立即验证
- 不要等多个 slice 攒在一起再验证

## 与其他方法的关系

- 是"低 churn 重构"的具体实施方式
- 配合"验证优先工作流"使用
- 配合"薄 skill + 厚文档"模式使用
